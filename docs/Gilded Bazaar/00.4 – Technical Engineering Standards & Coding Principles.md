# The Gilded Bazaar

## 00.4 – Technical Engineering Standards & Coding Principles

**Version:** 1.0
**Status:** Canonical Engineering Standard

---

# Purpose

This document defines the engineering standards for The Gilded Bazaar.

Its purpose is to ensure that every contributor—human or AI—implements features consistently, securely and maintainably.

These standards apply to every commit, pull request, feature branch and specification.

When implementation convenience conflicts with these standards, this document takes precedence.

---

# Engineering Philosophy

The codebase should optimise for:

* Readability
* Maintainability
* Predictability
* Testability
* Security
* Performance
* Scalability

The project should remain understandable by a new developer joining years after initial development.

# Architecture

Adopt a layered architecture.

```text
Presentation

↓

Feature Components

↓

Services

↓

Repositories

↓

Supabase / Edge Functions

↓

Database
```

Each layer has a single responsibility.

---

# Screen Responsibilities

Screens should:

* Render UI.
* Handle navigation.
* Coordinate feature components.

Screens must **not**:

* Execute SQL.
* Perform business calculations.
* Contain AI prompts.
* Validate purchases.
* Call RevenueCat directly.
* Implement pricing logic.

---

# Components

Components should be:

* Small
* Reusable
* Stateless whenever practical

Avoid monolithic components exceeding approximately 300–400 lines.

Complex behaviour should be extracted into hooks or services.

---

# Services

Services contain business logic.

Examples:

* MerchantService
* EconomyService
* PurchaseService
* NotificationService
* AIService

Services should expose clear public methods.

---

# Repository Layer

Repositories communicate with external systems.

Responsibilities include:

* Database access
* Storage
* Edge Functions
* External APIs

Repositories isolate infrastructure from business logic.

---

# Database Rules

Every table must include:

* Primary key
* Created timestamp
* Updated timestamp
* Ownership field where appropriate

Every table must implement Row Level Security.

Never rely solely on client-side validation.

---

# Security Principles

## Never Trust the Client

Clients are presentation layers.

They are not authoritative.

All sensitive operations occur server-side.

---

## API Keys

Secrets must never appear in:

* React Native code
* Git
* Logs
* Screenshots

Secrets belong in secure backend configuration only.

---

## Purchases

Credits must never be added client-side.

Workflow:

RevenueCat

↓

Backend verification

↓

Database update

↓

Realtime notification

---

## AI

Clients never call AI providers directly.

Workflow:

App

↓

Edge Function

↓

AI Provider

↓

Structured Response

↓

Validation

↓

Client

---

# State Management

Guidelines:

* Server state → React Query
* Local UI state → React hooks
* Global session state → Context
* Persistent preferences → Secure/local storage

Avoid unnecessary global state.

---

# Forms

All forms should use:

* React Hook Form
* Zod validation

Validation rules belong in schemas rather than components.

---

# TypeScript Standards

Strict mode enabled.

Avoid:

* any
* ts-ignore

Prefer:

* explicit interfaces
* discriminated unions
* shared types

Shared models belong in `/types`.

---

# Error Handling

Errors should:

* Be typed
* Be logged
* Provide recovery paths
* Never expose sensitive implementation details

User messages should remain friendly and actionable.

---

# Logging

Log:

* AI requests
* Purchases
* Authentication
* Permission changes
* Campaign events
* Errors
* Performance metrics

Never log:

* Secrets
* Access tokens
* Personal payment information

---

# AI Prompt Management

Prompts are code.

They must be:

* Version controlled
* Documented
* Tested
* Stored separately from UI

Prompts must never be embedded directly inside React components.

---

# JSON Contracts

All AI outputs must conform to documented schemas.

Never parse free-form responses when structured JSON is expected.

Validation failures should trigger retry or fallback behaviour.

---

# Testing Strategy

Every feature should include:

* Unit tests
* Integration tests
* UI tests where appropriate

Critical flows require end-to-end testing.

Examples:

* Authentication
* Campaign joining
* Purchase approval
* Credit deduction
* AI merchant generation

---

# Performance Standards

Target:

* 60 FPS navigation
* Fast cold start
* Efficient list rendering
* Lazy loading for large datasets
* Minimal unnecessary re-renders

Performance regressions should be investigated promptly.

---

# Accessibility Standards

Every screen should include:

* Accessible labels
* Semantic roles
* Dynamic text support
* Sufficient colour contrast
* Large touch targets

Accessibility is a release requirement.

---

# Offline Architecture

Offline-first where practical.

Users should continue to access cached campaign information.

Mutating actions should queue automatically and synchronise safely when connectivity returns.

Conflicts should be resolved predictably and logged.

---

# Git Workflow

Branch naming:

```text
feature/

fix/

refactor/

docs/

test/

chore/
```

Examples:

```text
feature/merchant-engine

fix/purchase-race-condition

docs/economy-specification
```

Commits should be small, focused and descriptive.

---

# Pull Request Standards

Every pull request should include:

* Purpose
* Related specification
* Testing completed
* Screenshots (if UI)
* Breaking changes
* Future considerations

---

# Documentation Requirements

Every implemented feature must reference its originating specification.

Specifications should be updated when implementation diverges for valid reasons.

The documentation is the source of truth.

---

# AI Coding Agent Guidelines

Before implementing code, every AI agent should:

1. Read the relevant specification.
2. Review related specifications.
3. Check applicable ADRs.
4. Confirm dependencies.
5. Follow established folder conventions.
6. Preserve architectural consistency.
7. Avoid introducing duplicate logic.

When uncertain, prefer consistency over novelty.

---

# Code Review Checklist

Before merging:

* Business logic separated from UI.
* Types documented.
* Tests passing.
* No hard-coded secrets.
* Accessibility maintained.
* Performance considered.
* Documentation updated.
* Acceptance criteria satisfied.

---

# Definition of Done

A feature is complete only when:

* Specification implemented.
* Tests written and passing.
* Documentation updated.
* Security reviewed.
* Accessibility verified.
* Performance acceptable.
* Code reviewed.
* Ready for release.
