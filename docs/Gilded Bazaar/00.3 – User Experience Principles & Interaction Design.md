# The Gilded Bazaar

## 00.3 – User Experience Principles & Interaction Design

**Version:** 1.0
**Status:** Living Document

---

# Purpose

This document defines the interaction philosophy of The Gilded Bazaar.

Where the Design Language defines how the application looks, this specification defines how it behaves.

Every screen, interaction, animation and workflow should follow these principles.

The goal is to create an experience that feels effortless, immersive and consistent across every part of the application.

---

# UX Vision

The Gilded Bazaar should feel less like navigating software and more like interacting with a living fantasy world.

Every action should feel:

* Predictable
* Calm
* Responsive
* Rewarding
* Immersive

Users should never feel overwhelmed by menus, settings or unnecessary complexity.

---

# Core UX Principles

## Principle 1 — Simplicity Before Features

Every screen should answer one primary question.

Examples:

* What can I buy?
* What has changed?
* What requires my approval?
* What is happening in my campaign?

If a screen attempts to answer multiple questions, it should be reconsidered.

---

## Principle 2 — Progressive Disclosure

Advanced functionality should appear only when it is needed.

New users should never be confronted with every available option.

Examples include:

* Advanced merchant controls hidden behind an "Advanced" section.
* AI generation settings collapsed by default.
* World simulation controls available only to authorised campaign roles.

---

## Principle 3 — Immersion Without Obstruction

Fantasy styling should enhance usability, never reduce it.

Decorative flourishes should support navigation rather than compete with it.

The interface should remain readable even after long sessions.

---

## Principle 4 — Immediate Feedback

Every action should produce visible feedback.

Examples:

* Purchase request submitted.
* Merchant restocked.
* Downtime project started.
* Credits deducted.
* AI generation completed.

Users should never wonder whether an action succeeded.

---

## Principle 5 — Respect User Time

The application exists to reduce administration.

Every interaction should remove friction.

Common actions should require minimal effort.

---

# Navigation Philosophy

Navigation should remain shallow.

Avoid deep menu hierarchies.

Recommended structure:

* Home
* Campaign
* Bazaar
* Backpack
* Journal

Secondary functionality should be accessed contextually rather than through additional navigation layers.

---

# Screen Hierarchy

Every screen should contain:

1. Clear title.
2. Context or status.
3. Primary action.
4. Supporting information.
5. Secondary actions.

Visual hierarchy should always make the primary action obvious.

---

# Information Density

Avoid overcrowding.

Prefer:

* Cards
* Sections
* Clear spacing
* Expandable panels

Rather than long uninterrupted lists.

---

# Primary Actions

Every screen should expose a single primary action.

Examples:

* Browse Shop
* Request Purchase
* Generate Merchant
* Begin Downtime
* Approve Requests

Primary actions should always be visually distinct.

---

# Secondary Actions

Secondary actions should never compete visually with the primary action.

Examples:

* Sort
* Filter
* Export
* Rename
* Archive

These should remain discoverable without dominating the interface.

---

# Empty States

Empty states should be helpful rather than blank.

Examples:

## Empty Shop

"No merchants have opened today."

Provide:

* Refresh
* Generate Merchant
* Learn More

---

## Empty Backpack

"Your journey has only just begun."

Provide:

* Browse Bazaar
* View Starter Equipment

---

## Empty Journal

"No adventures have been recorded yet."

Provide:

* Create First Entry
* Import Session Notes

Every empty state should encourage meaningful next actions.

---

# Loading States

Avoid blocking users whenever possible.

Preferred hierarchy:

1. Skeleton placeholders.
2. Progressive loading.
3. Inline activity indicators.
4. Full-screen loading only during initial authentication or essential setup.

---

# Error States

Errors should explain:

* What happened.
* Why it happened (where appropriate).
* What the user can do next.

Avoid technical jargon.

Example:

Instead of:

"RPC Error 500"

Display:

"The merchant ledger couldn't be reached. Please try again in a moment."

---

# Confirmation Philosophy

Do not confirm trivial actions.

Confirm only actions that:

* Spend credits.
* Delete data.
* Leave campaigns.
* Approve purchases.
* Remove merchants.
* Cancel downtime.

Low-risk actions should execute immediately.

---

# Undo Instead of Confirmation

Whenever possible:

Perform the action immediately.

Allow undo.

Examples:

* Archive journal entry.
* Remove favourite.
* Dismiss notification.

Undo is generally faster than asking for confirmation.

---

# Search

Search should be available throughout the application.

Searchable content includes:

* Items
* Merchants
* NPCs
* Quests
* Settlements
* Journal entries
* Campaign members

Search should be forgiving of spelling variations.

---

# Filtering

Filters should remain lightweight.

Common filters:

* Type
* Rarity
* Price
* Settlement
* Merchant
* Availability
* Pending Status

Filters should persist until intentionally cleared.

---

# Sorting

Provide sensible defaults.

Examples:

Items:

* Recommended
* Price
* Recently Added
* Alphabetical

Merchants:

* Distance
* Reputation
* Recently Updated

Journal:

* Most Recent
* Oldest
* Campaign Day

---

# Notifications

Notifications should feel like events within the world.

Examples:

"A travelling merchant has arrived."

"Your armour is ready for collection."

"The harvest festival begins tomorrow."

Notifications should reinforce immersion rather than simply announce system events.

---

# Haptic Feedback

Where supported:

Use subtle haptics for:

* Successful purchases.
* Approvals.
* Achievement milestones.
* Merchant generation.

Avoid excessive vibration.

---

# Gestures

Support natural mobile interactions.

Examples:

* Pull to refresh.
* Swipe to archive.
* Swipe for quick actions.
* Long press for contextual menus.

Never hide essential functionality behind gestures alone.

---

# Accessibility

Accessibility is mandatory.

Requirements include:

* Dynamic text support.
* High contrast compatibility.
* Screen reader labels.
* Keyboard navigation where applicable.
* Colour-independent indicators.
* Large touch targets.

The experience should remain enjoyable for all players.

---

# Offline Behaviour

Offline users should still be able to:

* Browse cached merchants.
* View inventories.
* Read campaign journals.
* Review NPC information.
* Track downtime.

Actions requiring synchronisation should queue automatically and sync when connectivity returns.

---

# AI Interactions

AI should remain largely invisible.

Users request outcomes, not prompts.

Examples:

Instead of:

"Generate Merchant"

Offer:

"Open a New Stall"

Instead of:

"Generate Quest"

Offer:

"Seek Adventure"

Technical implementation should remain behind the scenes.

---

# Performance Targets

Target interactions:

* Screen transitions under 300 ms.
* Cached content appears instantly.
* AI generation provides immediate progress feedback.
* Lists remain smooth at 60 FPS on supported devices.

Performance is a feature, not an optimisation.

---

# User Delight

Small moments create memorable experiences.

Examples:

* A subtle sparkle when a legendary item is discovered.
* A parchment unfurl animation for new quests.
* Coins gently scattering into the player's purse after a purchase.
* Candlelight flickering behind important announcements.

These moments should be rare enough to remain special.

---

# Definition of Success

A first-time user should understand how to complete common tasks without external guidance.

An experienced user should feel the application fades into the background, allowing them to focus on their campaign rather than the software.
