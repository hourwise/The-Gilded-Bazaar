# The Gilded Bazaar

## 00.6 – AI Design Philosophy & Prompt Engineering Standards

**Version:** 1.0
**Status:** Canonical AI Engineering Standard

---

# Purpose

This document defines the artificial intelligence philosophy, engineering standards and prompt architecture for The Gilded Bazaar.

Every AI-powered feature must comply with this specification.

No feature may bypass these standards.

---

# AI Vision

Artificial Intelligence should feel invisible.

Players should never think:

> "I'm talking to an AI."

Instead they should think:

> "This world feels alive."

AI is an engine powering the campaign behind the scenes.

It should reduce administration while increasing immersion.

---

# AI Principles

## Principle 1

AI assists creativity.

It never replaces it.

---

## Principle 2

The Dungeon Master always owns the world.

Generated content remains a suggestion until accepted.

---

## Principle 3

Structured data is more valuable than prose.

The application consumes structured information.

Narrative is generated afterwards.

---

## Principle 4

Consistency is more important than surprise.

Generated content must respect established lore.

---

## Principle 5

Every prompt should be deterministic whenever possible.

Repeated requests with identical inputs should produce compatible outputs.

---

# AI Responsibilities

Artificial Intelligence is responsible for:

* Merchant generation
* Shop generation
* Settlement flavour
* NPC personalities
* Rumours
* Downtime outcomes
* World events
* Notifications
* Quest inspiration
* Dialogue suggestions
* Item descriptions
* Economic recommendations

AI is **not** responsible for enforcing game rules.

---

# Human Responsibilities

Humans always control:

* Campaign canon
* Purchases
* Economy overrides
* World approval
* Player permissions
* Moderation decisions
* Premium purchases

The AI never overrides human authority.

---

# Provider Independence

The application must remain provider agnostic.

Supported providers may include:

* OpenAI
* Gemini
* Anthropic
* Local models

Changing provider should require configuration changes rather than architectural changes.

---

# AI Pipeline

Every request follows the same lifecycle.

```text
User Action
      │
      ▼
Business Service
      │
      ▼
AI Request Builder
      │
      ▼
Context Assembly
      │
      ▼
Prompt Template
      │
      ▼
AI Provider
      │
      ▼
JSON Validation
      │
      ▼
Business Rules Validation
      │
      ▼
Persistence
      │
      ▼
User Interface
```

No screen communicates directly with an AI provider.

---

# Prompt Architecture

Every prompt consists of independent layers.

## System Layer

Defines permanent behaviour.

Never references campaign-specific data.

Example responsibilities:

* Tone
* Safety
* JSON format
* Constraints

---

## Context Layer

Provides campaign-specific information.

Examples:

* Settlement
* Merchant
* Economy
* Player level
* Existing world state
* Historical events

---

## Instruction Layer

Defines the requested task.

Examples:

Generate a merchant.

Generate a tavern menu.

Create rumours.

Update inventory.

---

## Output Contract

Defines required JSON schema.

No free-form output should be accepted.

---

# Prompt Versioning

Every prompt must include:

* Identifier
* Version
* Last modified date
* Owning specification
* Expected schema version

Prompt changes should be tracked like code changes.

---

# JSON Contracts

Every AI response must validate against a documented schema.

Example:

```json
{
  "merchant": {},
  "inventory": [],
  "personality": {},
  "rumours": []
}
```

Invalid responses are rejected.

---

# Validation Pipeline

Every AI response passes through:

1. JSON validation.
2. Schema validation.
3. Business rule validation.
4. Safety validation.
5. Persistence.

Failure at any stage prevents publication.

---

# Context Management

The AI should receive only the context required for the task.

Avoid unnecessary data.

Benefits include:

* Lower cost.
* Faster responses.
* Reduced hallucinations.
* Greater consistency.

---

# World Consistency

Before generation, AI receives:

* Existing campaign facts.
* Settlement profile.
* Merchant history.
* Current world events.
* Economy state.
* Relevant NPC relationships.

The AI should extend the world rather than reinvent it.

---

# Hallucination Prevention

Never ask AI to invent unrestricted content.

Instead:

Provide constraints.

Provide examples.

Provide world state.

Provide schema.

Request only the missing information.

---

# Retry Strategy

If validation fails:

Attempt structured regeneration.

If repeated failures occur:

Fallback to deterministic templates.

The application should never display malformed AI output.

---

# Caching

Generated content should be cached where appropriate.

Suitable candidates include:

* Merchant biographies.
* Item descriptions.
* Settlement flavour.
* Tavern menus.
* Rumours.

Avoid regenerating identical content unnecessarily.

---

# Cost Optimisation

AI is a premium resource.

Optimisation strategies include:

* Context minimisation.
* Prompt reuse.
* Result caching.
* Batched requests.
* Tiered models.

Simple tasks should not consume expensive models.

---

# Prompt Library

Every prompt belongs in a central Prompt Library.

Each prompt should define:

* Purpose
* Inputs
* Outputs
* Schema
* Dependencies
* Example request
* Example response
* Version history

No production prompt should exist solely inside application code.

---

# AI Safety

The platform must prevent generation of:

* Offensive content.
* Harassment.
* Illegal content.
* Personal data.
* Copyright infringement.
* Unsafe instructions.

Safety checks should occur both before and after generation where appropriate.

---

# AI Transparency

Users should know when content is AI-assisted.

However, the experience should remain immersive.

Example:

"Generated Merchant"

is preferable to

"ChatGPT Response."

---

# Offline Behaviour

Offline users should continue to access previously generated content.

Generation requests queue until connectivity returns.

---

# Performance Targets

Target generation times:

Simple generation

< 2 seconds

Complex generation

< 6 seconds

Long-form generation

Progress indicators required.

---

# AI Testing

Every prompt requires testing.

Test categories include:

* Schema compliance.
* Lore consistency.
* Edge cases.
* Empty context.
* Invalid inputs.
* Large campaigns.
* Low-resource settlements.

Prompt quality should improve through iterative refinement.

---

# Future AI Capabilities

Potential future systems include:

* Voice NPCs.
* Dynamic conversations.
* Procedural newspapers.
* Merchant negotiations.
* Regional dialects.
* Campaign recaps.
* Automatic session summaries.
* Predictive economy balancing.
* Player-specific storytelling.

These should remain compatible with the core AI architecture.

---

# Definition of Success

The player experiences a believable, coherent world.

The Dungeon Master spends less time preparing content.

The AI behaves consistently regardless of provider.

Prompt improvements can be made without changing application code.
